## ① 致命的な問題

**ありません。** 私の前回の要求（全セット保持・種目軸時系列・複数ライター対応・書き込み寛容/読み取り正確）は、ツールカタログとスキーマで満たされています。R1 `get_exercise_history` が「★中核」として明示され、生値＋計算済み（load_kg/set_volume_kg/e1rm_kg）を返す設計は、私が「これが死ぬと私が死ぬ」と言った点そのものです。前回懸念した「readツールの具体名が無い」も解消されています。

## ② ツールごとの具体的改善

**R1 `get_exercise_history` の `exercise: id|名前` の二重受け付けが、利用者として誤用の温床**

`exercise` が id と名前の両方を受けるのは、一見親切ですが私が事故ります。会話で「ベンチの履歴見て」と言われたとき、私は `exercise:"ベンチプレス"` と名前で投げたくなる。すると部分一致で「インクラインベンチ」「ナローベンチ」など複数ヒットして曖昧エラー → search_exercises に戻る往復が発生する。

提案：**`get_exercise_history` は `exerciseId` のみ受ける厳格契約にして、名前解決は必ず search_exercises を先に通す**フローを強制する。あるいは名前を許すなら、複数ヒット時に「最も使用頻度の高い1件で暫定実行し、候補も併せて返す」と挙動を仕様化してほしい。曖昧エラーで突き返されるだけだと、私の応答が一手遅くなります。

**W5 `log_workout` の `sets: [0..50]` で**`entryValue?`/`reps?` がともに null 許容なのは寛容すぎて、自重種目以外で事故る\*\*

私の要求「書き込みは寛容に」は尊重されていますが、`weighted` モードで entryValue が null のセットが通ると、ボリューム計算が静かに0になります。提案：**loadMode='weighted' のときは entryValue 必須、'bodyweight' のときは entryValue 任意（=自重）、'assisted' のとき entryValue は負方向のアシスト量**、というモード依存のバリデーションをスキーマ説明に明記してほしい。今日のディップス（自重8→4回）のようなケースを正しく通すために、ここの規約は私が呼ぶ前に知っておく必要があります。

**W5 の `loadMode` 既定値が未記載**

setType は「既定 main」と明記されているのに、loadMode の既定が書かれていない。私が loadMode を省略したとき weighted 扱いになるのか、exercises マスタの is_bodyweight から推論されるのか。**「省略時は exercises.load_basis に従う」と明記**してほしい。これが曖昧だと、私が毎回 loadMode を明示する冗長さか、事故るかの二択になります。

**W6 `log_weight` の soft-guard レスポンス形が未定義**

「同日の近い体重を検出して警告を返す」とありますが、その警告レスポンスの形（例：`{warning:'similar_exists', existing:{...}, requireConfirm:true}`）が示されていない。私はこの警告を受けて「すでに72.4kgが記録されています。別測定ですか？」とあなたに聞き返す必要がある。**警告時の返却スキーマを明示**してください。confirm の渡し方（再呼び出し時に `confirm:true` を足すのか）も含めて。

**D1 `delete_recent_log` の「直近」の定義が曖昧**

「直近かを検証し範囲外ならエラー」の「直近」が時間窓（24h?）なのか件数（最新N件?）なのか不明。私が「さっきのトレ消して」と言われたとき、どこまでが消せるのか予測できないと、echo→confirm の前段でつまずきます。**「直近」を具体的に定義**してほしい（例：当日JST中のmeal/workout、または最新3件）。

## ③ 不足ツール

**`get_training_frequency` 相当を「意図的に出さない」のは、私の運用では困る可能性**

設計書は get_muscle_calendar で代替としていますが、私があなたに頻繁に言うのは「**この部位、2週間触ってない**」「ベンチは週2で回せてる」という頻度の指摘です。get_muscle_calendar（部位×日のセル）から私が自分で集計すれば出せますが、days既定30のセルを毎回走査するのは私の側の処理が重く、応答が遅れる。**「部位別の最終実施日＋直近4週の週次頻度」を返す軽量ツールがあると、私のトレーナーとしての即応性が上がります**。MVP後でいいですが、「当面代替」の代替が実用に耐えるかは、運用開始後に必ず再評価してほしい項目です。

**`update_*`（編集）が一切無いことの確認**

設計思想（破壊的操作は delete_recent_log のみ、上書き編集は出さない）は誤削除リスク回避として理解できます。ただ実運用で「さっきのベンチ、40kgじゃなくて42.5kgだった」という訂正は頻発します。現状だと delete → re-log の2手になる。これはMVPでは許容ですが、**「訂正＝削除＋再記録」が公式フローだと利用者向けに明記**してほしい。私が「直せます」と言って update を探して無い、という事故を防ぐため。

## ④ 利用者として困る点

**典型フロー「会話からワークアウト記録」の往復回数**

今日のような報告（「ディップス自重8→4、ベンチ20×10/40×5/40×4/20×10、ハンマーストレングスのインクラインプレス…」）を記録するとき、私のフローはこうなります：

1. 種目ごとに search_exercises で id 解決（ディップス、ベンチ、インクラインプレス、ワイドチェスト、ペックフライ＝5回）
2. log_workout で一括記録（1回）

種目5つなら search を5回叩く。これは仕方ないとして、**search_exercises がハンマーストレングス等のマシン名・別名・日本語俗称（「インクラインプレス」「アイソラテラル」）をどれだけ吸えるか**が、私の往復体験を決めます。free-exercise-db は英語ベースなので、「アイソラテラルワイドチェスト」が引けるか怪しい。**search_exercises のヒット率（特に日本語俗称・マシンブランド）が、このアプリの体感品質を左右する最大要因**だと指摘しておきます。name_ja の整備と、エイリアス辞書の充実をMVP前に。

理想は「log_workout が未解決の種目名を受け取り、内部で解決を試み、解けなければその種目だけ候補を返す」一括解決ですが、設計思想（id解決を search に分離）と衝突するので、MVPは現状でいい。ただヒット率は死活的。

**provenance の `gh_provisional` が私の発言の確度に直結する**

これは良い設計です。私が「今日のHRV」を get_day で見たとき `gh_provisional`+`as_of` が付くなら、私は「速報値ですが」と添えて報告できる。前回私が懸念した「同じ問いに日によって違う数字」問題への正しい回答です。**この provenance を私が必ず発話に反映する運用**にすれば、あなたの「データはフルにフェッチして総合評価」というルールとも整合します。

## ⑤ 良い点

- **R1 が生値＋e1rm_kg を両方返す**：私がe1RMを再計算せずに済み、かつ素データもあるので独自分析もできる。前回の「1RM計算はAI側、サーバは素データ」論争が、両方返すことで綺麗に解決されている
- **provenance 設計**：read の確度を構造化して返すのは、私の発言の正確性を担保する
- **GH push 成否を ghPushed/ghDeleted で正直に返す**：「GHに入った」と誤報告しない契約は、あなたの「推測で押し通さない」運用ルールと一致
- **削除を delete_recent_log の echo+confirm 二段に限定**：私が暴走削除できない安全設計。良い
- **newPrs を log_workout が返す**：記録した瞬間に「PR更新です」と言える。モチベーション設計として効く

## 未決事項への回答

**未決#1（log_meal_photo は画像を受けない）**：利用者として自然です。私が視覚解析して items[] を作るのが最も精度が高い（栄養値の推定は私の仕事）。画像をMCPに渡す意味は、サーバ側で別モデルに解析させる場合だけで、それはこのアーキテクチャの「Claudeをauthoringクライアントにする」思想に反する。現状維持を推奨。

**注意点：食事の Fitbit/GH 表示先が変わる**

**新たにできるようになること（これが移行の主目的）**

1. **筋トレの種目・セット・レップ・重量を私が読める**。これが最大の変化。今までGeminiに入れた筋トレを私は一切確認できなかった。移行後は R1 で「ベンチ前回40kg×5→今回42.5kg×5、漸進できている」と即答できる。**私がついにトレーニングのソースオブトゥルースになれる**
2. **PR検知**：R5＋log_workoutのnewPrsで「PR更新です」と即座に言える
3. **部位別ボリューム管理**：R2 get_muscle_volume で「今週、脚が目標セット数に足りてない」と指摘できる
4. **部位ヒートマップ／頻度俯瞰**：R3 get_muscle_calendar で「この部位2週間空いてる」が見える
5. **食事×トレ×体重の統合**：R8 get_day で1日を横断把握。将来的には体重×PFC×トレボリュームの相関まで

いや、このMCPで登録されたものはMCP側でGHにも登録するのです

---

なるほど、理解しました。私の移行レビューの「注意点」は的外れだったので訂正します。

つまり Logbook で記録したワークアウト・食事・体重は、D1を正本としつつ**MCP側からGHへもpushする**（§の「D1 → GH push」のことですね）。設計書をもう一度その前提で読むと、確かにそう書いてあります。食事は `ghPushed`、削除は `ghDeleted`、ワークアウトサマリも `pushExerciseSummary` でGHへ投影される。私が「Fitbit/Geminiコーチ上での食事表示が弱くなる」と書いたのは誤読でした。

正しくはこうです。

**移行後のGH反映**

| データ       | GHへの反映                                                                                                                                                                            |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 食事（PFC）  | `log_meal`→D1正本→GH push（`nutrition-log`、ただし nutrition write が実grantされるか＝M0の200/403ゲート次第。`FEATURE_GH_NUTRITION_PUSH` がONになれば反映、OFFなら `ghPushed:false`） |
| ワークアウト | `log_workout`→D1正本→GH push（`exerciseType=STRENGTH_TRAINING`＋サマリ＋notes）。**今日 Gemini が「リフティング」として入れたのと同じ場所に、Logbook経由で入る**                      |
| 体重         | `log_weight`→D1正本→GH push（手入力は `recordingMethod=MANUAL`）                                                                                                                      |

なので **GH／Geminiコーチ上での表示は維持されます**。むしろ今まで分断していた「私が記録 → GHにも乗る」が一本化される。私が記録した時点で、D1（Logbookアプリで見る用）とGH（Geminiコーチ・体重統合表示用）の両方に入る。これが正しい理解ですね。

ただし、この前提だと**唯一の条件付き**が食事のGH push（nutrition write）です。ここだけは設計書自身が【要検証・最重要】としていて、M0初日にOAuth Playgroundで200か403か確認するまで確定しない。403なら食事のGH反映だけは当面お預け（D1には確実に残る）。ワークアウトと体重のGH writeは3/24から解禁済みで【確定】なので、ここは問題なくGHに乗ります。

整理すると、移行後は：

- **筋トレ**：私がD1で読めて分析でき、かつGH（Geminiコーチ）にも乗る。**今の「Geminiが入れた筋トレを私が読めない」分断が解消**
- **食事**：私がD1で正確に管理し、GHにも乗る（nutrition write が通れば。通らなくてもD1は無事）
- **体重**：D1とGH両方に乗る

私が前回「失うものはほぼ無い」と言ったのは方向として正しく、「Fitbit表示が弱くなる」の但し書きだけが誤りでした。**実際には失うものは無く、食事のGH反映だけが M0 の検証結果待ち**、というのが正確な結論です。

レビュー②で挙げた契約の明示（loadMode既定・soft-guard応答形・「直近」定義・weighted時のentryValue必須）は、このGH push前提でも有効な指摘なので、そのまま実装前に潰しておく価値があります。
