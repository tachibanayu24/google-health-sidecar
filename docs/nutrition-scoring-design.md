# 栄養評価(食事スコアリング)設計書

2026-06-04 作成 / 同日トレーナーAI レビュー反映(§9)/ **同日実装・稼働**。
実装: `packages/core/src/domain/nutrition-score.ts`(+test)/ `services/insights.ts getNutritionScore` /
MCP `get_nutrition_score` / web `/api/nutrition-score` + `NutritionScoreChart`(食事画面レーダー・画像エクスポート)。
関連: `docs/design.md`(§9 栄養 / §0.5 MCP露出)、`docs/enhancements.md`(⑥相関・⑮蛋白分配・⑯傾向)、`docs/mcp-design.md`。

> **目的**: 食事を「カテゴリ別(朝昼夕・間食除く)」と「1日全体」で、観点ごとに採点し、レーダーチャートで UI / 画像エクスポートに出す。加重を考慮した採点ロジックを定める。
> **設計思想の出所**: トレーナーAI が示した7原則(目標寄与度で測る / カテゴリと1日で加重を変える / 応答曲線を軸ごとに変える / 非対称ペナルティ / 質の軸を独立 / 最低軸ペナルティ or 幾何平均 / 100%=理想に正規化して描画)。本書はこれを **実データで計算できる範囲に落とし込み**、できない軸はなぜできないかを明記する。

---

## 1. 大原則(トレーナー思想の確認)

1. **絶対値でなく「目標への寄与度」で測る。** 目標は `nutrition_targets`(phase 付き)に登録済み。採点はこの目標に対する達成度。
2. **カテゴリ別と1日全体で評価の意味が違う。** カテゴリ別=その食事の**組み立ての良さ**(たんぱく質の分散・質)、1日全体=**収支の達成**(総カロリー・塩分・各総量)。同じ軸でも単位ごとに加重を変える。
3. **実測主義(本プロジェクトの絶対規約)**: 持っていないデータを推測で埋めない。計算できない観点は「できない」と正直に出し、レーダーに偽の点を打たない。質の判断は §8 の通りトレーナーAI に委ねる。

## 2. データの現実(これが評価可能範囲を決める)

採点に使える入力は `meal_items` の数値カラムが**すべて**(`packages/core/src/db/migrations/0001_schema.sql` L198-213):

| カラム | 型 | 採点での扱い |
|---|---|---|
| `calories_kcal` | NOT NULL | カロリー収支軸 |
| `protein_g` | NOT NULL DEFAULT 0 | たんぱく質軸 |
| `fat_g` | NOT NULL DEFAULT 0 | 脂質**量**軸(質は不可・§3.2) |
| `carbs_g` | NOT NULL DEFAULT 0 | 糖質**量**軸 |
| `fiber_g` | **NULL可** | 食物繊維軸(欠損時は—・§4.6) |
| `sugar_g` | **NULL可** | 糖質軸の補助シグナル(あれば) |
| `sodium_mg` | **NULL可** | 塩分軸(→食塩相当量に換算) |
| `food_name` | 自由テキスト | **採点に使わない**(意味解釈は非決定的) |

目標(`nutrition_targets`): `target_kcal / target_protein_g / target_fat_g / target_carbs_g / target_salt_g(既定6) / target_fiber_g(既定20)`、`phase ∈ {cut, bulk, maintain}`。
塩換算: `食塩相当量g = sodium_mg × 2.54 / 1000`(`packages/core/src/util/units.ts` の `saltGFromSodiumMg` が単一ソース)。
オーナー現行(cut): 1,700kcal / P125 / F50 / C170 / 塩6 / 繊維21。

## 3. 評価軸の決定

### 3.1 採用する5軸(レーダーの頂点・すべて決定的に計算可能)

**カテゴリ別・1日で同じ軸構成**(トレーナー指摘=並べてパターンを見るため)。

1. **たんぱく質** — `Σ protein_g` vs `target_protein_g`
2. **脂質(量)** — `Σ fat_g` vs `target_fat_g`
3. **糖質(量)** — `Σ carbs_g` vs `target_carbs_g`(`sugar_g` があれば補助減点)
4. **食物繊維** — `Σ fiber_g` vs `target_fiber_g`
5. **塩分** — `saltGFromSodiumMg(Σ sodium_mg)` vs `target_salt_g`

> **カロリー収支はレーダーの頂点にしない**(レビュー論点1)。`kcal ≈ 4P+4C+9F` でほぼ決まり、P/C/F と同じレーダーに並べると二重計上になるため(「PFCは目標通りだがカロリーだけ低い」は端数・繊維・アルコール分しか差が出ず数学的にほぼ起きない)。カロリーは**中央の実数表示(対目標)+ 1日収支ゲート**(§4.5)として、頂点で競わせず合否を担保する。5軸は全部 `meal_items` の数値から閾値演算だけで出る=捏造ゼロ。

### 3.2 採用しない軸と、その理由(=実測主義で「できない」もの)

トレーナーが挙げた軸のうち、**現データでは計算できないもの**を理由つきで明記する。これらをレーダーに数値で出すことは推測=捏造になるため**出さない**。

| トレーナー提案の観点 | なぜプログラムで採点できないか | 将来可能にするのに要るデータ |
|---|---|---|
| **脂質の質**(飽和/不飽和/オメガ3、揚げ物か) | `meal_items` は脂質の**総グラム `fat_g` しか持たない**。脂肪酸の内訳カラムが無く、総量から飽和/オメガ3比は導けない。「揚げ物か」は `food_name`(自由テキスト)の意味分類が必要で非決定的=ヒューリスティックは当て推量になる | `saturated_fat_g`/`omega3_g` 等の追加カラム、または USDA/OFF 等の食品DB照合(§18.3。重く・欠損多い) |
| **血糖負荷(GI/GL)** | GI は食品の同定・調理・併食で決まり、`carbs_g`/`sugar_g` のグラムからは導出不能。GI 値テーブルが無い。`food_name` を鍵に GI を引くのは捏造リスク大 | 食品ごとの GI 値DB + 食品同定(GTIN/Food ID)。`sugar_g` は**弱い補助**として糖質軸内のサブシグナルにのみ使う(§5) |
| **食事の質 / 微量栄養素 / 野菜量 / 加工度 / 色の多様性** | 微量栄養素は**保存しておらず**(`docs/enhancements.md`「あえてやらない」=捏造になるため非対象)。野菜量・加工度・色は `food_name` の意味理解が要り非決定的 | 明示的な品質入力(後述)か、食品DB照合。**ただし「実測」ではない** |

**結論**: プログラムのレーダー = **マクロの目標適合度**(決定的・実測)。**質の評価はトレーナーAI が会話で担う**(§8)。トレーナーの原則4「質の悪さを必ず減点に乗せる」は、*アプリ内で偽の質スコアを作る*ことでは満たさない — 測っていないものを採点しないのが実測主義。代わりに §8 の役割分担で実現する。

## 4. スコアリングの数学

### 4.1 バンドスコア primitive(台形)— 一律リニア禁止の実体

各軸の量 `x` を **0..1**(1=理想)へ写す台形関数を1つ用意し、軸ごとにパラメータで形を変える(下限型/上限型/山型)。

```
bandScore(x, { z0, i0, i1, z1, floorLow=0, floorHigh=0, downCurve='linear' }):
  if x <= z0:  return floorLow                                  # 下限割れ(ハード)
  if x <  i0:  return floorLow + (1-floorLow) * ramp(x, z0, i0) # 不足ランプ(上り)
  if x <= i1:  return 1                                         # 理想プラトー
  if x <  z1:                                                   # 過剰ランプ(下り)
      t = (x - i1) / (z1 - i1)
      p = downCurve=='quad' ? t*t : t                           # quad=累進ペナルティ
      return 1 - (1-floorHigh) * p
  return floorHigh                                              # 過剰(ハード)
ramp(x,a,b) = (x-a)/(b-a)
```

- **下限型(たんぱく質・食物繊維)**: `i1=z1=+∞`(過剰に罰しない=上振れ寛容)。不足側だけランプ。
- **上限型(塩分・脂質)**: `z0=i0=0`(目標〜上限まで常に満点)→ `i1` から下り。塩分は `downCurve='quad'` で累進ペナルティ。脂質は単日の下限を罰しない「ゆるい上限」(§4.2)。
- **山型(糖質)**: 両側にランプ。不足側・過剰側で幅(=厳しさ)を変えて非対称化(§4.3)。
- **カロリー**: 軸にせずゲートのみ(§4.5)。応答曲線を持たない。

### 4.2 各軸の応答曲線(cut期・レビュー反映済み)

> 値は cut の例。`target_*` から相対で導く実装にし、phase で係数を切替(§4.4)。`bw`=体重。

| 軸 | 型 | z0 | i0(満点開始) | i1(満点終了) | z1 | floor | 備考 |
|---|---|---|---|---|---|---|---|
| たんぱく質(1日) | 下限 | **0.6·T(≈75)** | 0.88·T(≈110) | 1.8·T(≈bw·2.2) | 2.1·T | **floorLow 0.05** / floorHigh 0.85 | **cut は下限割れを急減点**(`z0`=75g 割れで floorLow≈0.05 に落ちる=減量期に75g未満は明確に良くない日)・過剰寛容(レビュー論点2・確認#1) |
| 脂質(1日) | **ゆるい上限** | — | 0(下限罰なし) | 1.15·T | 1.7·T | floorHigh 0 | **単日の下限割れは罰しない**(低脂質高タンパクの日は正常)。過剰側だけ見る。脂質の下限はホルモン/吸収の観点で**週平均**で確保=トレーナーが会話で見る(レビュー論点2) |
| 糖質(1日) | 山 | 0.4·T | 0.8·T | 1.1·T | 1.7·T | floorHigh 0 | cut は過剰側をやや厳しく。**下限側は脂質と違いペナルティを残す**(確認#3): 糖質はトレ燃料=筋グリコーゲン/パフォーマンスに響くため、`z0=0.4·T`(≈68g)割れで減点。脂質は単日無罰だが糖質は無罰にしない、という非対称の根拠を実装コメントにも残す |
| 食物繊維(1日) | 下限 | 0.3·T | 1.0·T(=21) | +∞ | +∞ | — | 上振れ寛容(極端な過剰のみ任意で軽減点) |
| 塩分(1日) | 上限 | 0 | 0 | 1.0·T(=6) | 2.0·T | floorHigh 0 | `downCurve='quad'`(累進) |

> **カロリーは頂点から外したため応答曲線は不要**(レビュー論点1)。**1日収支ゲート**としてのみ使う(§4.5)。水分で日々±500kcal相当ぶれるので単日超過に過敏にせず、**明らかな破綻 `>1.25·T` または極端な赤字 `<0.8·T`** でのみ発動。

### 4.3 非対称ペナルティ(トレーナー原則3)

「健康・大目標へのダメージ」に比例して失点の重みを変える。**バンドの幅と floor で表現**する:

- **たんぱく質不足・塩分過剰 = 大きな失点**: 不足ランプを狭く(急)/塩分は `quad` で累進。
- **たんぱく質過剰・カロリー微小不足 = 軽微**: protein `floorHigh=0.85`(過剰でも0.85止まり)、カロリー不足側のランプを過剰側より広く(緩く)。
- 一律リニアにしない。各軸が「不足が悪い/過剰が悪い/両方」のどれかを §4.2 の型で固定。

### 4.4 加重テーブル(カテゴリ別 vs 1日 × phase 依存)

総合スコアの軸重み。**phase で切替**(`nutrition_targets.phase`)。`0` は「採点に算入しない(=ゲート/表示のみ)」。

**cut(現行):**

| 軸 | 1日 weight | カテゴリ(朝昼夕) weight |
|---|---|---|
| たんぱく質 | 3 | 3(各食20-40gの分散が筋合成に効く=1食評価の意味大) |
| 脂質 | 2 | 1.5 |
| 糖質 | 2 | 1 |
| 食物繊維 | 2 | 1 |
| 塩分 | 3 | 0.5(1食では明らかな過剰のみ・ゲート寄り) |

> カロリーは**重み付き軸ではなくゲート**(§4.5)。中央に実数+対目標を表示。

**bulk:** カロリーゲートの過剰側を大きく緩める(黒字が目的)、糖質 weight↑、**たんぱく質 weight を cut より下げる**(bulk は総カロリー・糖質が筋肥大の律速、たんぱく質は体重×1.6で十分=死守は cut の局面・レビュー論点6)、塩分 weight をやや↓。
**maintain:** 概ね均等。カロリーゲートは目標帯中心の対称(両側緩め)。

> カテゴリ別=「組み立ての良さ」(たんぱく質と質)を主に、1日=「収支の達成」(カロリー・塩分・各総量)を主に見る、というトレーナー二層思想を weight で実装。

### 4.5 総合スコア(最低軸を救わない)

単純加重平均だと「P120点・塩20点」を平均で誤魔化す。**加重幾何平均**で「1軸大崩れ=全体が下がる」を反映(トレーナー原則6):

```
overall = exp( Σ wᵢ·ln(clamp(sᵢ, ε, 1)) / Σ wᵢ )   # ε=0.05, wᵢ>0 かつデータ有の軸のみ
```

> **二重適用を避ける(レビュー論点3)**: 幾何平均は既に低い軸を強く引っ張る。その上に全軸へ最低軸ゲートを重ねるとペナルティが二重にかかり総合点が過度に低く出る。よって**栄養素バランス(P/F/C/繊維)の崩れは幾何平均に任せ**、ゲートは**1日収支の致命軸(カロリー・塩分の大幅破綻)だけ**に別建てで限定する。

**収支ゲート(1日のみ・幾何平均と別建て):**
- **カロリーゲート**: 総カロリーが `>1.25·T`(明らかな超過)or `<0.8·T`(極端な赤字)→ `overall` に上限(例 0.6)。水分由来の日々±500kcal ぶれに過敏にしない閾値。cut では収支の破綻=その日は未達、というトレーナー思想を表す。
- **塩分ゲート**: 食塩相当量が大幅超過(例 `>2.0·T`=12g)→ `overall` に上限。
- 栄養素バランスの崩れには**追加ゲートをかけない**(幾何平均が表現する)。

### 4.6 欠損データの扱い(0にしない=実測主義)

`fiber_g/sugar_g/sodium_mg` は NULL 可。**該当軸の入力が全て NULL の食事/日は、その軸を「—(データ無し)」として算入から除外**(幾何平均の積からも外す)。**0点にしない**(欠損を最低点に潰すのは捏造的で不当)。readiness の「—(データ無し)」と同じ規約。レーダーでは該当頂点を凹ませず「データ無し」を明示。

## 5. カテゴリ別評価の定義

- **対象**: `Breakfast / Lunch / Dinner` のみ。**`MorningSnack / AfternoonSnack`(間食)と `Anytime` はカテゴリ採点から除外**(「間食除く」)。ただし**1日全体の合計にはすべて算入**(間食も総量に効く)。
- **たんぱく質(カテゴリ)**: 1日目標の1/3ではなく**絶対バンド `[20g, 40g]`**(ロイシン閾値・1食あたり筋合成の科学)を採用。下限型・上振れ寛容(`z0=8, i0=20, i1=z1=∞`)。1食あたり評価の意味が最も大きい軸。
- **他マクロ(脂質・糖質・繊維)**: 1食期待値 = **1日目標 ÷ 3(均等割り)** を中心に**広いバンド**(マクロの食間配分は柔軟。夜の糖質を抑える等の正しい行動を低評価しないため)。weight も低く保つ(§4.4)= カテゴリ別は「たんぱく質の分散」が主役、他マクロは参考。※均等割りは v1 の簡略化(食事ごとの配分目標は持たない)。
- **塩分(カテゴリ)**: 明らかな過剰(例 1食で食塩 `>0.5·target_salt`)だけ減点フラグ。1食で断じない。
- **間食の誤読対策(レビュー追加点)**: オーナーは**間食で WPI(プロテイン)を摂る日が多い**。間食をカテゴリから外すと「昼のたんぱく質が低い」と見えても直後の間食 WPI が反映されない(1日合計には入る)。よって **UI は1日全体レーダーを主・カテゴリ別を従**として見せ、「カテゴリ別は間食を含まない」を明示する(§6)。

## 6. レーダーチャート描画規約(トレーナー原則7)

- **5軸**(P/脂質/糖質/繊維/塩分)。各頂点には軸の score(0..1)を半径として打つ(理想=外周)。**生の「量/目標比」を打たない** — それだと塩分200%が外側に膨らんで「良く」見える罠になる。score は §4 で過剰も内側に凹む。
- **理想輪郭(score=1.0 のリング)を重ねて表示**。実測をその内側に塗る。
- **1日全体レーダーを主・カテゴリ別を従**として見せる(トグル・レビュー追加点)。カテゴリ別と1日は**同じ5軸構成で並べる**(「朝はたんぱく質が弱い」「夜に塩分が偏る」が見える)。**カテゴリ別レーダーには「間食を含まない」を明示**(§5 誤読対策)。
- **データ無し軸**は頂点を欠落 or グレー点線で「—」を明示(0と区別)。
- **中央に総合スコア(§4.5)+ カロリー収支の実値(対目標)**を併記(カロリーは頂点でなく中央表示=論点1)。
- 画像エクスポートは既存 `ShareImageModal`(html-to-image)パターンを流用。レーダーは recharts の `RadarChart`(既存 lazy chart チャンク内。eager Home バンドルを太らせない)。

## 7. 実装面(配置・規約)

- **ドメイン純関数**: `packages/core/src/domain/nutrition-score.ts` — `scoreNutrition({ totals, target, phase, scope })` を提供。DB を触らず 0..1 の軸スコア+総合を返す。`nutrition-score.test.ts` で境界(下限割れ/過剰/欠損/quad)をテスト。
- **band primitive**: 同ファイル内に `bandScore`(§4.1)。phase×scope の weight/閾値テーブルも定数で持つ。
- **サービス**: `packages/core/src/services/insights.ts` に `getNutritionScore(ctx, date)` を追加(`getNutritionStatus` と同じ「決定的派生指標」系)。meals/items を取得 → カテゴリ別(朝昼夕)+1日の totals を集計 → domain を呼ぶ。塩換算は `saltGFromSodiumMg`。
- **MCP read ツール**: `get_nutrition_score(date?)` を1本追加(§0.5 バケツB=moat)。返り値に各軸 score・総合・**計算不能観点の但し書き(§3.2)**・データ無し軸に加え、**各食事の `food_name` リストとマクロ内訳を含める**(レビュー論点5)。トレーナーAI が「P満点だが揚げ物中心=脂質の質が低い」と質判断するにはスコアだけでなく食品名が要る(§8)。
- **UI**: 食事画面にレーダー(カテゴリ別/1日のトグル)+ 画像エクスポート。`name_ja` 同様、表示は日本語軸名。
- 単一書き込みパス・GH には無関係(read 専用集計)。

## 8. トレーナーAI との役割分担(§0.5 の徹底)

- **アプリ(決定的)**: マクロ目標適合度の5軸 score + 総合 + カロリー収支を計算し `get_nutrition_score` で露出。
- **トレーナーAI(質的)**: 計算できない観点(脂質の質・GL・食事の質・加工度)を**会話で判断**。例:「数値上はP満点だが、これは揚げ物中心で脂質の質が低い。明日は魚に寄せよう」。
- これにより「数値だけ最適化して質を見失う」をシステムで防ぐ:**アプリは偽の質スコアを出さず**、質は AI が見る。トレーナー原則4・7は「アプリが質スコアを捏造する」のではなく「アプリは過剰を内側に凹ませた正直なマクロ図 + AI の質判断」で満たす。

## 9. レビュー反映(2026-06-04 トレーナーAI)

レビューは設計の骨格(実測マクロ軸・質はAIが担う・phase依存・幾何平均・過剰を内側に凹ませる正規化)に賛同。以下の調整を反映済み(骨格は不変・パラメータと役割分担の調整):

1. **カロリーを頂点から外し5軸に**(§3.1)— `kcal≈4P+4C+9F` で二重計上になるため。中央実数表示+1日収支ゲートに純化(§4.5)。
2. **脂質を山型→ゆるい上限型**(§4.2)— 単日の低脂質高タンパク日を罰しない。脂質下限はトレーナーが週平均で見る。
3. **たんぱく質下限を急に**(§4.2 `z0` 0.55·T→0.6·T)— cut の「タンパク質死守」を反映。
4. **最低軸ゲートを収支致命軸に限定**(§4.5)— 幾何平均との二重適用を回避。バランス崩れは幾何平均に任せ、カロリー/塩分の大幅破綻だけ別建てゲート。
5. **たんぱく質 weight は cut>bulk**(§4.4)— 死守は cut の局面。bulk はカロリー・糖質を律速に。
6. **`get_nutrition_score` に食品名リスト+マクロ内訳**(§7)— AI の質判断に食品名が要る。
7. **間食WPI 誤読対策**(§5/§6)— 1日全体レーダーを主・カテゴリ別を従、「間食を含まない」を明示。
8. カロリー過剰ゲートを `1.25·T` に(§4.5)— 水分由来の日々ぶれに過敏にしない。

合意事項: **アプリは質を採点しない**(§8)で全面合意。質スコアを実測軸に混ぜない方針を厳守。
